① ブラウザ開発者ツールのパネル

② console オブジェクトのメソッド

console の主要メソッド一覧

// 基本的な出力
console.log('通常のログ', { key: 'value' });
console.info('情報メッセージ');
console.warn('警告メッセージ');
console.error('エラーメッセージ');

// テーブル形式で表示
const users = [
  { name: 'Alice', age: 25 },
  { name: 'Bob',   age: 30 }
];
console.table(users); // 表形式で見やすく表示

// グループ化
console.group('ユーザー処理');
  console.log('処理開始');
  console.log('処理中...');
console.groupEnd();

// 折りたたみグループ
console.groupCollapsed('詳細ログ（折りたたみ）');
  console.log('詳細1');
console.groupEnd();

// アサーション（条件が false のときだけ出力）
console.assert(1 === 2, '1 は 2 ではありません'); // 出力される
console.assert(1 === 1, 'これは出力されない');    // 出力されない

// カウント
console.count('click'); // click: 1
console.count('click'); // click: 2
console.countReset('click');

// タイマー
console.time('処理時間');
// ... 処理 ...
console.timeLog('処理時間');  // 途中経過
console.timeEnd('処理時間'); // 処理時間: 12.34ms
📌 試験頻出：console.assert の動作 console.assert(condition, message) は 条件が false のときだけエラーメッセージを出力します。 true のときは何も出力しません。 テスト・デバッグ時の簡易アサーションとして使います。

③ ブレークポイントと debugger 文

ブレークポイントの種類

種類設定方法用途
行ブレークポイントSources パネルで行番号をクリック特定の行で一時停止
条件付きブレークポイント行番号を右クリック → 条件を入力条件が true のときだけ停止
DOM 変更ブレークポイントElements で要素右クリック → Break onDOM 変更時に停止
イベントリスナーブレークポイントSources → Event Listener Breakpoints特定イベント発生時に停止
例外ブレークポイントSources の ⏸ アイコン例外発生時に自動停止
debugger 文コードに debugger; を記述プログラムから停止を指示
 
// debugger 文：開発者ツールが開いているときだけ停止
function calculateTotal(items) {
  debugger; // ← ここで実行が一時停止する

  let total = 0;
  for (const item of items) {
    total += item.price;
  }
  return total;
}

// ステップ実行のショートカット
// F8  / Resume       → 次のブレークポイントまで実行
// F10 / Step over    → 現在の行を実行して次の行へ（関数の中に入らない）
// F11 / Step into    → 関数の中に入る
// Shift+F11 / Step out → 現在の関数を抜ける

// 条件付きデバッグ（本番コードでよく使うパターン）
const DEBUG = true;
if (DEBUG) {
  console.log('デバッグ情報:', data);
}
⚠️ debugger 文を本番コードに残さない debugger 文は開発者ツールが開いていると実行を停止させます。 本番環境に残すとユーザーの操作を妨げる可能性があります。 ビルドツール（webpack など）で自動削除するか、必ず手動で削除しましょう。

④ パフォーマンス計測

console.time() と performance.now()

// console.time() / timeEnd()：シンプルな計測
console.time('ループ処理');
for (let i = 0; i < 1000000; i++) { /* 処理 */ }
console.timeEnd('ループ処理'); // → ループ処理: 12.345ms

// performance.now()：高精度タイムスタンプ（マイクロ秒精度）
const start = performance.now();

// 計測したい処理
const arr = [];
for (let i = 0; i < 100000; i++) arr.push(i);

const end = performance.now();
console.log(`処理時間: ${(end - start).toFixed(3)}ms`);

// performance.mark() / measure()：詳細な計測
performance.mark('start');
// ... 処理A ...
performance.mark('middle');
// ... 処理B ...
performance.mark('end');

performance.measure('処理A', 'start', 'middle');
performance.measure('処理B', 'middle', 'end');

const measures = performance.getEntriesByType('measure');
measures.forEach(m => console.log(m.name, m.duration.toFixed(3) + 'ms'));
方法精度特徴用途
console.time()msシンプル・手軽簡易計測
performance.now()μs（マイクロ秒）高精度・差分計算精密計測・ベンチマーク
performance.mark()μs複数ポイントを記録詳細なプロファイリング
DevTools Performancems視覚的・CPU/メモリも総合的なパフォーマンス分析

⑤ エラーの種類と try-catch-finally

JavaScriptの主要エラー型

エラー型発生原因例
SyntaxError構文エラー（パース時）let x = ;
ReferenceError未定義の変数を参照console.log(undeclared)
TypeError型が不正な操作null.property
RangeError値が有効範囲外new Array(-1)
URIErrorURI 関数の不正使用decodeURIComponent('%')
EvalErroreval() 関連（現代では稀）
 
// try-catch-finally の基本構造
try {
  // エラーが発生する可能性のある処理
  const data = JSON.parse('invalid json'); // SyntaxError
} catch (error) {
  // エラーが発生したときの処理
  console.error(error.name);    // → 'SyntaxError'
  console.error(error.message); // → エラーメッセージ
  console.error(error.stack);   // → スタックトレース
} finally {
  // エラーの有無に関わらず必ず実行
  console.log('クリーンアップ処理');
}

// エラーの種類で分岐
try {
  riskyOperation();
} catch (e) {
  if (e instanceof TypeError) {
    console.error('型エラー:', e.message);
  } else if (e instanceof RangeError) {
    console.error('範囲エラー:', e.message);
  } else {
    throw e; // 処理できないエラーは再スロー
  }
}

// カスタムエラー
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = 'ValidationError';
    this.field = field;
  }
}
throw new ValidationError('必須項目です', 'email');
📌 試験頻出：finally ブロックの特性 finally ブロックは try / catch のどちらが実行されても必ず実行されます。 try 内で return しても finally は実行されます。 リソースの解放・クリーンアップ処理に使います。

📋 デモ④：デバッグ ベストプラクティス ステップガイド

各ステップをクリックして展開し、実践的なデバッグ手順を学びましょう。 すべてのステップを確認するとデバッグの全体像が掴めます。
1
🔍 エラーメッセージを正確に読む

コンソールのエラーメッセージにはエラーの種類・メッセージ・ファイル名・行番号が含まれます。

Uncaught TypeError: Cannot read properties of null (reading 'textContent')
at updateUI (app.js:42:18)
at init (app.js:10:3)

→ app.js の 42行目で null の textContent にアクセスしようとしている。
→ getElementById が null を返している可能性が高い（IDのタイポなど）。

2
🖨️ console.log で変数の値を確認する

問題が起きている箇所の前後に console.log を追加して、変数の値を確認します。

// ❌ 問題のあるコード
function updateUI(data) {
document.getElementById('result').textContent = data.value;
}

// ✅ デバッグ用 console.log を追加
function updateUI(data) {
console.log('data:', data); // データを確認
console.log('el:', document.getElementById('result')); // 要素を確認
document.getElementById('result').textContent = data.value;
}
3
⏸️ ブレークポイントでステップ実行する

console.log だけでは追いきれない場合は、Sources パネルでブレークポイントを設定してステップ実行します。

  • F8 / Resume：次のブレークポイントまで実行
  • F10 / Step over：現在の行を実行（関数に入らない）
  • F11 / Step into：関数の中に入る
  • Shift+F11 / Step out：現在の関数を抜ける

右側の Scope パネルで現在のスコープの変数値を確認できます。
Watch パネルで特定の式をウォッチできます。

4
🛡️ try-catch でエラーを適切に処理する

外部APIの呼び出しやユーザー入力の処理など、エラーが予想される箇所は try-catch で囲みます。

5
📊 パフォーマンスを計測して最適化する

処理が遅い場合は performance.now() で計測してボトルネックを特定します。

// どの処理が遅いか特定する
const t0 = performance.now();
heavyOperation1();
const t1 = performance.now();
heavyOperation2();
const t2 = performance.now();
console.log
(`op1: ${(t1-t0).toFixed(2)}ms, op2: ${(t2-t1).toFixed(2)}ms`);

よくある最適化：
・DOM操作をまとめる（DocumentFragment）
・ループ内の DOM 取得をキャッシュする
・不要なイベントリスナーを削除する
