1. Web Workers

Web Workers はメインスレッドとは別のスレッドで JavaScript を実行します。重い計算処理をワーカーに移すことで、UI がブロックされずにスムーズに動作します。

Web Workers の基本 API 
// ── メインスレッド側 ──

// Worker の作成（別ファイルを指定）
const worker = new Worker('worker.js');

// Worker にメッセージを送信
worker.postMessage({ type: 'start', data: 1000000 });

// Worker からメッセージを受信
worker.onmessage = (event) => {
  console.log('Worker から受信:', event.data);
};

// Worker のエラーを捕捉
worker.onerror = (error) => {
  console.log('Worker エラー:', error.message);
};

// Worker を終了（メインスレッドから）
worker.terminate();

// ── worker.js 側 ──

// メインスレッドからメッセージを受信
self.onmessage = (event) => {
  const { type, data } = event.data;
  if (type === 'start') {
    // 重い計算処理
    let result = 0;
    for (let i = 0; i < data; i++) {
      result += Math.sqrt(i);
    }
    // メインスレッドに結果を送信
    self.postMessage({ type: 'result', result });
  }
};

// Worker 自身を終了
self.close();

// ── Worker でアクセスできないもの ──
// DOM（document, window, document.getElementById など）
// window オブジェクト（alert, confirm など）
// localStorage / sessionStorage
// ── Worker でアクセスできるもの ──
// XMLHttpRequest / fetch
// setTimeout / setInterval
// IndexedDB
// WebSocket
// importScripts()

📌 試験頻出：Web Workers の制約
  • Worker は DOM にアクセスできない（document・window・alert 等は使用不可）
  • localStorage・sessionStorage にもアクセスできない
  • メインスレッドとの通信は postMessage() と onmessage のみ
  • データはコピー（構造化複製）されて渡される（参照共有ではない）
  • 例外: ArrayBuffer は Transferable として転送（コピーなし）できる
  • Worker 内で別の Worker を作成することも可能（Nested Worker）

2. High Resolution Time（performance API）

performance API の使い方 
// ── performance.now() ──
// ページ読み込み開始からの経過時間（ミリ秒、小数点以下あり）
const t0 = performance.now();
// ... 処理 ...
const t1 = performance.now();
console.log(`処理時間: ${t1 - t0} ms`); // 例: 12.345678 ms

// Date.now() との違い
// Date.now()        → Unix タイムスタンプ（ms整数）、精度低め
// performance.now() → ページ起動からの相対時間（μs精度）、単調増加

// ── performance.mark() ── カスタムマーカーを記録
performance.mark('fetch-start');
// ... fetch 処理 ...
performance.mark('fetch-end');

// ── performance.measure() ── 2つのマーク間の時間を計測
performance.measure(
  'fetch-duration',  // 計測名
  'fetch-start',     // 開始マーク
  'fetch-end'        // 終了マーク
);

// ── 記録を取得 ──
const marks    = performance.getEntriesByType('mark');
const measures = performance.getEntriesByType('measure');
const entry    = performance.getEntriesByName('fetch-duration')[0];
console.log(entry.duration);  // 計測時間（ms）
console.log(entry.startTime); // 開始時刻（ms）

// ── 記録をクリア ──
performance.clearMarks();
performance.clearMeasures();

// ── PerformanceObserver（非同期で監視）──
const observer = new PerformanceObserver((list) => {
  list.getEntries().forEach(entry => {
    console.log(entry.name, entry.duration);
  });
});
observer.observe({ entryTypes: ['measure'] });
📌 試験頻出：performance.now() の特性
  • マイクロ秒精度（小数点以下あり）。Date.now() はミリ秒整数
  • 単調増加（システム時刻変更の影響を受けない）
  • ページの navigationStart からの相対時間（Unix タイムスタンプではない）
  • セキュリティのため、精度が意図的に下げられることがある（Spectre 対策）
  • performance.mark() と performance.measure() でユーザー定義の計測ポイントを設定できる

3. Page Visibility API

Page Visibility API の使い方 
// ── document.visibilityState ──
// 'visible'  → ページが表示されている
// 'hidden'   → ページが非表示（別タブ・最小化・ロック画面等）
// 'prerender' → ページがバックグラウンドでプリレンダリング中（非推奨）

console.log(document.visibilityState); // 'visible' or 'hidden'
console.log(document.hidden);          // true: 非表示, false: 表示中

// ── visibilitychange イベント ──
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    // タブが非表示になった → アニメーション停止・通信削減
    pauseAnimation();
    stopPolling();
    console.log('ページが非表示になりました');
  } else {
    // タブが表示された → アニメーション再開・データ更新
    resumeAnimation();
    startPolling();
    console.log('ページが表示されました');
  }
});

// ── 活用例: 動画の自動停止 ──
document.addEventListener('visibilitychange', () => {
  if (document.hidden) {
    video.pause();
  } else {
    video.play();
  }
});

// ── 活用例: requestAnimationFrame との組み合わせ ──
function animate() {
  if (!document.hidden) {
    // 表示中のみ描画
    drawFrame();
  }
  requestAnimationFrame(animate);
}
📌 試験頻出：document.visibilityState の値
  • 'visible' — ページが少なくとも部分的に表示されている
  • 'hidden' — 別タブに切り替え・ウィンドウ最小化・画面ロック時
  • document.hidden は visibilityState === 'hidden' のとき true
  • requestAnimationFrame はタブが非表示のとき自動停止するが、setInterval は停止しない → Page Visibility API と組み合わせて制御する

4. Navigation Timing API

Navigation Timing API の使い方 
// ── performance.timing（旧 API）──
const t = performance.timing;

// 各フェーズの時間を計算
const dns      = t.domainLookupEnd  - t.domainLookupStart;  // DNS 解決時間
const tcp      = t.connectEnd       - t.connectStart;       // TCP 接続時間
const ttfb     = t.responseStart    - t.requestStart;       // Time to First Byte
const download = t.responseEnd      - t.responseStart;      // ダウンロード時間
const domParse = t.domInteractive   - t.responseEnd;        // DOM パース時間
const domReady = t.domContentLoadedEventEnd - t.navigationStart; // DOMContentLoaded まで
const loadTime = t.loadEventEnd     - t.navigationStart;    // load イベントまで

// ── Navigation Timing Level 2（新 API）──
const [navEntry] = performance.getEntriesByType('navigation');
console.log(navEntry.domainLookupEnd  - navEntry.domainLookupStart); // DNS
console.log(navEntry.connectEnd       - navEntry.connectStart);      // TCP
console.log(navEntry.responseStart    - navEntry.requestStart);      // TTFB
console.log(navEntry.domContentLoadedEventEnd);                       // DCL
console.log(navEntry.loadEventEnd);                                    // Load
console.log(navEntry.type);   // 'navigate' / 'reload' / 'back_forward'
console.log(navEntry.transferSize); // 転送バイト数

5. オンライン状態の検出

オンライン/オフライン検出 
// ── navigator.onLine ──
console.log(navigator.onLine); // true: オンライン, false: オフライン

// ── online / offline イベント ──
window.addEventListener('online', () => {
  console.log('オンラインになりました');
  // キューに溜めたリクエストを送信
  flushRequestQueue();
});

window.addEventListener('offline', () => {
  console.log('オフラインになりました');
  // リクエストをキューに保存
  saveToQueue(pendingRequest);
});

// ── 注意点 ──
// navigator.onLine が true でも実際に通信できるとは限らない
// （ルーターには繋がっているがインターネット不通の場合など）
// 確実な確認には fetch で実際にリクエストを送る
async function checkConnectivity() {
  try {
    await fetch('/ping', { method: 'HEAD', cache: 'no-store' });
    return true;
  } catch {
    return false;
  }
}

🧩 演習問題

4-6 パフォーマンスとオンラインの理解を確認しましょう。

QUESTION 1 / 6
Web Workers でアクセスできないものはどれですか？
✅ 正解：C（document.getElementById()）
Web Workers は DOM にアクセスできません。document・window・alert() などは使用不可です。
A ○ XMLHttpRequest は Worker 内で使用可能
B ○ IndexedDB は Worker 内で使用可能
C ✗ document.getElementById() は DOM 操作 → Worker では使用不可
D ○ setTimeout() は Worker 内で使用可能
また localStorage・sessionStorage も Worker では使用できません。
QUESTION 2 / 6
performance.now() と Date.now() の違いについて正しい説明はどれですか？
✅ 正解：B
performance.now() の特性：
• ページの navigationStart からの相対時間（Unix タイムスタンプではない）
• マイクロ秒精度（小数点以下あり）
• 単調増加（システム時刻変更の影響を受けない）
Date.now() は Unix タイムスタンプ（1970年1月1日からのミリ秒整数）を返します。
パフォーマンス計測には performance.now() が適しています。
QUESTION 3 / 6
document.visibilityState が取りうる値として正しい組み合わせはどれですか？
✅ 正解：B（'visible' と 'hidden'）
document.visibilityState の値：
• 'visible': ページが少なくとも部分的に表示されている
• 'hidden': 別タブに切り替え・ウィンドウ最小化・画面ロック時
（仕様上 'prerender' もありますが、現代のブラウザではほぼ使われません）
document.hidden プロパティは visibilityState === 'hidden' のとき true になります。
QUESTION 4 / 6
以下のコードで、Worker からメインスレッドにメッセージを送信する正しい方法はどれですか？
// worker.js 内のコード self.onmessage = (event) => { const result = heavyCalculation(event.data); // ← ここでメインスレッドに結果を送信 };
✅ 正解：C（self.postMessage(result)）
Worker 内でメインスレッドにメッセージを送信するには self.postMessage() を使います。
• self は Worker のグローバルオブジェクト（WorkerGlobalScope）
• window は Worker 内では使えない
• parent は Nested Worker での親 Worker への参照（通常は使わない）
• worker はメインスレッド側の変数名（Worker 内からは参照できない）
メインスレッド側では worker.onmessage で受信します。
QUESTION 5 / 6（複数選択）
Navigation Timing API で計測できる時間として正しいものをすべて選んでください。
※ 正しいものをすべてクリックして「確認」を押してください
✅ 正解：A・B・D・E
Navigation Timing API で計測できるフェーズ：
A ○ DNS 解決: domainLookupEnd - domainLookupStart
B ○ TCP 接続: connectEnd - connectStart
C ✗ ユーザー操作の時間は Navigation Timing では計測できない（User Timing API を使う）
D ○ DOM パース: domInteractive - responseEnd
E ○ TTFB: responseStart - requestStart（最初の1バイトを受信するまでの時間）
