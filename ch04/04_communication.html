1. WebSocket

WebSocket はサーバーとクライアント間で双方向・全二重通信を実現するプロトコルです。HTTP のハンドシェイクで接続を確立した後、ws:// または wss:// プロトコルで持続的な接続を維持します。

WebSocket の基本 API 
// WebSocket 接続の確立
const ws = new WebSocket('wss://example.com/socket');
// ws:// → 非暗号化、wss:// → TLS 暗号化（推奨）

// ── readyState の値 ──
// WebSocket.CONNECTING = 0  接続中
// WebSocket.OPEN       = 1  接続済み（送受信可能）
// WebSocket.CLOSING    = 2  切断中
// WebSocket.CLOSED     = 3  切断済み

// ── イベントハンドラ ──
ws.onopen = (event) => {
  console.log('接続完了 readyState:', ws.readyState); // → 1
  ws.send('Hello Server!');          // 文字列を送信
  ws.send(JSON.stringify({ type: 'ping' })); // JSON を送信
};

ws.onmessage = (event) => {
  console.log('受信:', event.data);   // 文字列 or Blob or ArrayBuffer
  const data = JSON.parse(event.data);
};

ws.onclose = (event) => {
  console.log('切断 code:', event.code);   // 1000=正常, 1006=異常切断
  console.log('reason:', event.reason);
  console.log('wasClean:', event.wasClean);
};

ws.onerror = (event) => {
  console.log('エラー発生');
};

// ── メッセージ送信 ──
ws.send('text message');           // 文字列
ws.send(new ArrayBuffer(8));      // バイナリ
ws.send(new Blob(['data']));      // Blob

// ── 接続を閉じる ──
ws.close();                        // 正常切断
ws.close(1000, '正常終了');       // コードと理由を指定

// ── バッファリングされた送信データ量 ──
console.log(ws.bufferedAmount);    // 未送信のバイト数

// ── 受信データの型を指定 ──
ws.binaryType = 'arraybuffer';    // 'blob'（デフォルト）or 'arraybuffer'
readyState 値定数名状態send() の可否
0CONNECTING接続中（ハンドシェイク中）不可
1OPEN接続済み（通信可能）可能
2CLOSING切断処理中不可
3CLOSED切断済み不可
📌 試験頻出：WebSocket の readyState
  • CONNECTING=0、OPEN=1、CLOSING=2、CLOSED=3 の4値を必ず覚える
  • send() が呼べるのは readyState === 1（OPEN）のときのみ
  • HTTP とは異なり、接続を維持したまま双方向にメッセージを送受信できる
  • wss:// は TLS で暗号化された WebSocket（HTTPS の ws 版）
  • HTTP のアップグレードリクエストで接続を確立する（Upgrade: websocket ヘッダー）

2. XMLHttpRequest（XHR）

XMLHttpRequest は HTTP リクエストを非同期で送受信する API です。readyState プロパティで処理の進行状態を管理します。

XMLHttpRequest の基本パターン 
const xhr = new XMLHttpRequest();

// ── readyState の値 ──
// UNSENT           = 0  open() 未呼び出し
// OPENED           = 1  open() 呼び出し済み
// HEADERS_RECEIVED = 2  レスポンスヘッダー受信済み
// LOADING          = 3  レスポンスボディ受信中
// DONE             = 4  通信完了

xhr.open('GET', 'https://api.example.com/data', true);
// open(method, url, async=true, user, password)

// リクエストヘッダーの設定（open の後、send の前）
xhr.setRequestHeader('Content-Type', 'application/json');
xhr.setRequestHeader('Authorization', 'Bearer token123');

// レスポンスの型を指定
xhr.responseType = 'json'; // '' / 'text' / 'json' / 'blob' / 'arraybuffer' / 'document'

// タイムアウト設定（ms）
xhr.timeout = 5000;

// 状態変化イベント
xhr.onreadystatechange = () => {
  if (xhr.readyState === 4) {           // DONE
    if (xhr.status === 200) {
      console.log(xhr.response);        // レスポンスデータ
      console.log(xhr.responseText);    // テキスト形式
      console.log(xhr.responseURL);     // 最終 URL（リダイレクト後）
      console.log(xhr.getResponseHeader('Content-Type'));
    }
  }
};

// 進捗イベント
xhr.onprogress = (e) => {
  if (e.lengthComputable) {
    console.log(`${e.loaded} / ${e.total} bytes`);
  }
};

xhr.onload    = () => console.log('完了');
xhr.onerror   = () => console.log('ネットワークエラー');
xhr.ontimeout = () => console.log('タイムアウト');
xhr.onabort   = () => console.log('中断');

// リクエスト送信
xhr.send();                            // GET の場合は引数なし
xhr.send(JSON.stringify({ key: 'val' })); // POST の場合はボディを渡す

// 中断
xhr.abort();
readyState 値定数名状態
0UNSENTopen() が呼ばれていない初期状態
1OPENEDopen() が呼ばれた。send() 可能
2HEADERS_RECEIVEDsend() が呼ばれ、レスポンスヘッダーを受信済み
3LOADINGレスポンスボディを受信中
4DONE通信完了（成功・失敗問わず）
📌 試験頻出：XHR の readyState
  • UNSENT=0、OPENED=1、HEADERS_RECEIVED=2、LOADING=3、DONE=4
  • 実際のレスポンス処理は readyState === 4 かつ status === 200 で行う
  • onreadystatechange は readyState が変わるたびに呼ばれる（0→1→2→3→4）
  • responseType = 'json' を設定すると response が自動的にパースされたオブジェクトになる

3. Fetch API

Fetch API の基本と応用 
// ── 基本的な GET リクエスト ──
fetch('https://api.example.com/users')
  .then(response => {
    console.log(response.status);     // 200, 404, 500 など
    console.log(response.ok);         // true: 200〜299, false: それ以外
    console.log(response.statusText);  // 'OK', 'Not Found' など
    console.log(response.headers.get('Content-Type'));
    if (!response.ok) throw new Error(`HTTP error: ${response.status}`);
    return response.json();           // Promise<Object>
  })
  .then(data => console.log(data))
  .catch(err => console.log('エラー:', err));

// ── POST リクエスト（JSON 送信）──
fetch('https://api.example.com/users', {
  method:  'POST',
  headers: { 'Content-Type': 'application/json' },
  body:    JSON.stringify({ name: 'Alice', age: 30 }),
  mode:    'cors',        // 'cors' / 'no-cors' / 'same-origin'
  credentials: 'include', // 'omit' / 'same-origin' / 'include'
  cache:   'no-cache',    // 'default' / 'no-cache' / 'reload' / 'force-cache'
});

// ── Response のボディ読み取りメソッド ──
response.json();        // Promise<Object> — JSON パース
response.text();        // Promise<string> — テキスト
response.blob();        // Promise<Blob>   — バイナリ
response.arrayBuffer(); // Promise<ArrayBuffer>
response.formData();   // Promise<FormData>

// ── async/await で書く ──
async function fetchData() {
  try {
    const res  = await fetch('https://api.example.com/data');
    if (!res.ok) throw new Error(`${res.status}`);
    const data = await res.json();
    return data;
  } catch (err) {
    console.log('fetch エラー:', err);
  }
}

// ── AbortController でキャンセル ──
const controller = new AbortController();
fetch(url, { signal: controller.signal });
controller.abort(); // リクエストをキャンセル

4. Server-Sent Events（SSE）

Server-Sent Events はサーバーからクライアントへの一方向リアルタイム通信を実現します。HTTP 接続を維持したまま、サーバーがイベントをプッシュします。

EventSource（SSE）の使い方 
// EventSource の作成
const es = new EventSource('/events');
// withCredentials オプション
const es2 = new EventSource('/events', { withCredentials: true });

// ── readyState ──
// EventSource.CONNECTING = 0  接続中（再接続中も含む）
// EventSource.OPEN       = 1  接続済み
// EventSource.CLOSED     = 2  切断済み

// ── イベントハンドラ ──
es.onopen = () => console.log('SSE 接続完了');

es.onmessage = (e) => {
  console.log('受信:', e.data);    // data フィールドの値
  console.log('id:', e.lastEventId); // イベント ID
};

// カスタムイベント名で受信
es.addEventListener('update', (e) => {
  console.log('update イベント:', e.data);
});

es.onerror = (e) => {
  if (es.readyState === EventSource.CLOSED) {
    console.log('接続が切断されました');
  } else {
    console.log('エラー。自動再接続します...');
  }
};

// 接続を閉じる（自動再接続しない）
es.close();

// ── SSE のイベントストリーム形式（サーバー側）──
// Content-Type: text/event-stream
//
// : コメント行（クライアントには届かない）
//
// id: 1
// event: update
// data: {"message": "Hello"}
//
// data: 複数行の
// data: データも可能
//
// retry: 3000
// data: 再接続間隔を3秒に設定
比較項目Server-Sent EventsWebSocket
通信方向サーバー → クライアント（一方向）双方向（全二重）
プロトコルHTTP（通常の HTTP 接続を維持）ws:// または wss://（独自プロトコル）
自動再接続あり（ブラウザが自動で再接続）なし（手動で実装が必要）
データ形式テキストのみテキスト + バイナリ
用途通知・ライブフィード・株価更新チャット・ゲーム・リアルタイム協調
HTTP/2 対応効率的に多重化できる別途接続が必要

4 非同期処理パターン比較デモ

同じ処理（3ステップの非同期タスク）をコールバック・Promise・async/await で実装した場合の違いを確認できます。

コールバック

// コールバックパターン（コールバック地獄）
function step1(callback) {
  setTimeout(() => {
    console.log('Step 1: ユーザー取得');
    callback(null, { id: 1, name: 'Alice' });
  }, 500);
}
function step2(user, callback) {
  setTimeout(() => {

function step3(orders, callback) {
  setTimeout(() => {
    console.log('Step 3: 配送情報取得');
    callback(null, { status: 'shipped' });
  }, 500);

// ネストが深くなる（コールバック地獄）
step1((err, user) => {
  if (err) return console.log('エラー:', err);
  step2(user, (err, orders) => {
    if (err) return console.log('エラー:', err);

      if (err) return console.log('エラー:', err);
      console.log('完了:', delivery);

function step1() {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      console.log('Step 1: ユーザー取得');

  });
}
function step2(user) {
  return new Promise((resolve) => {
    setTimeout(() => {
      console.log('Step 2: 注文取得');
      resolve([{ id: 101, item: 'Book' }]);
    }, 500);

}
function step3(orders) {
  return new Promise((resolve) => {
    setTimeout(() => {
      console.log('Step 3: 配送情報取得');
      resolve({ status: 'shipped' });
    }, 500);
  });

step1()

// async/await パターン（同期的な書き方）
function step1() {
  return new Promise(resolve =>
    setTimeout(() => resolve({ id: 1, name: 'Alice' }), 500)
  );
}
function step2(user) {
  return new Promise(resolve =>
    setTimeout(() => resolve([{ id: 101, item: 'Book' }]), 500)
  );

function step3(orders) {
  return new Promise(resolve =>
    setTimeout(() => resolve({ status: 'shipped' }), 500)

QUESTION 1 / 6

      
WebSocket の readyState が 1 のとき、その状態を表す定数名はどれですか？

      

        
        
        
        
      

      

        ✅ 正解：B（OPEN）

        WebSocket の readyState は4値です：

        • 0: CONNECTING — 接続中（ハンドシェイク中）

        • 1: OPEN — 接続済み。send() で送信可能

        • 2: CLOSING — 切断処理中

        • 3: CLOSED — 切断済み

        send() を呼べるのは readyState === 1（OPEN）のときだけです。
      

    


    
    

      
QUESTION 2 / 6

      
XMLHttpRequest の readyState が 4（DONE）になったとき、必ずしも通信が成功したとは言えない理由として正しいものはどれですか？

      

        
        
        
        
      

      

        ✅ 正解：C

        readyState === 4（DONE）は「通信処理が完了した」ことを示すだけです。HTTP ステータスコードが 200（成功）でも 404（Not Found）でも 500（Server Error）でも、readyState は 4 になります。

        実際の成功判定には status === 200 または status >= 200 && status < 300 を確認する必要があります。

        ネットワークエラーの場合は onerror イベントが発火します。
      

    


    
    

      
QUESTION 3 / 6

      
Server-Sent Events（SSE）と WebSocket の違いについて正しい説明はどれですか？

      

        
        
        
        
      

      

        ✅ 正解：B

        SSE（Server-Sent Events）の特徴：

        • サーバー → クライアントの一方向通信（クライアントからサーバーへは通常の HTTP リクエストを使う）

        • 自動再接続：接続が切れるとブラウザが自動的に再接続する

        • HTTP プロトコル上で動作（ws:// ではない）

        • テキストデータのみ（バイナリ非対応）

        WebSocket は双方向・全二重通信が可能で、バイナリも送受信できます。
      

    


    
    

      
QUESTION 4 / 6

      
Fetch API について誤っている説明はどれですか？

      

        
        
        
        
      

      

        ✅ 正解（誤り）：C

        Fetch API の重要な特性：

        A ○ fetch() は Promise を返します。

        B ○ HTTP ステータスが 404・500 でも Promise は fulfilled（解決）になります。response.ok が false になるだけです。

        C ✗（誤り） ネットワークエラー（DNS 解決失敗・接続拒否など）の場合は Promise が rejected（拒否）になります。

        D ○ response.ok は HTTP ステータスが 200〜299 のとき true です。

        ✅ 正解：A・B・D

        WebSocket の send() で送信できるのは：

        A ○ 文字列（String）— テキストフレームとして送信

        B ○ ArrayBuffer — バイナリフレームとして送信

        また TypedArray（Uint8Array など）も送信可能です。
