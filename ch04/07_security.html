1. 同一オリジンポリシー（Same-Origin Policy）

同一オリジンポリシーは、あるオリジンから読み込まれたドキュメントやスクリプトが、別のオリジンのリソースにアクセスすることを制限するブラウザのセキュリティ機構です。

オリジンは プロトコル（スキーム）・ホスト（ドメイン）・ポート番号 の3要素で構成されます。これら3つがすべて一致する場合のみ「同一オリジン」となります。

オリジンの構成と判定例 
// オリジン = プロトコル + ホスト + ポート
// https://example.com:443/path?query#hash
//  ↑プロトコル  ↑ホスト  ↑ポート  ↑パス以降はオリジンに含まれない

// ── 基準 URL: https://example.com/page.html ──

// ✅ 同一オリジン
// https://example.com/other.html     → プロトコル・ホスト・ポート同じ
// https://example.com/dir/page.html  → パスが違うだけ

// ❌ 異なるオリジン（クロスオリジン）
// http://example.com/page.html       → プロトコルが違う (http vs https)
// https://other.com/page.html        → ホストが違う
// https://sub.example.com/page.html  → サブドメインも別オリジン
// https://example.com:8080/page.html → ポートが違う

// JavaScript でオリジンを取得
console.log(location.origin);
// → 'https://example.com'

// URL オブジェクトでオリジンを解析
const url = new URL('https://api.example.com:8080/data');
console.log(url.protocol); // 'https:'
console.log(url.hostname); // 'api.example.com'
console.log(url.port);     // '8080'
console.log(url.origin);   // 'https://api.example.com:8080'

// 同一オリジンポリシーが制限するもの
// - XMLHttpRequest / fetch によるクロスオリジンリクエスト
// - iframe 内のクロスオリジンページへの DOM アクセス
// - クロスオリジンの Cookie・localStorage へのアクセス
// ※ <img>・<script>・<link> などの埋め込みは許可される
比較対象 URL基準: https://example.com/理由
https://example.com/other.html✅ 同一オリジンパスのみ異なる
http://example.com/❌ 異なるプロトコルが違う（http vs https）
https://www.example.com/❌ 異なるホストが違う（サブドメイン）
https://example.com:8080/❌ 異なるポートが違う（443 vs 8080）
https://other.com/❌ 異なるホストが違う
📌 試験頻出：同一オリジンの判定基準
  • オリジン = プロトコル + ホスト + ポート の3要素すべてが一致すること
  • パス・クエリ・フラグメントはオリジンに含まれない
  • http:// と https:// は別オリジン（プロトコルが違う）
  • example.com と sub.example.com は別オリジン（サブドメインも別）
  • HTTPS のデフォルトポートは 443、HTTP は 80。明示しない場合はデフォルトポートが使われる

2. CORS（Cross-Origin Resource Sharing）

CORS は、クロスオリジンリクエストを安全に許可するための仕組みです。サーバーが特定のレスポンスヘッダーを返すことで、ブラウザがクロスオリジンアクセスを許可します。

CORS の主要ヘッダー 
// ── サーバー側のレスポンスヘッダー ──

// 許可するオリジン（* = すべて許可）
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Origin: *  // ※ credentials と併用不可

// 許可するメソッド（プリフライト応答で使用）
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS

// 許可するリクエストヘッダー
Access-Control-Allow-Headers: Content-Type, Authorization, X-Custom-Header

// 認証情報（Cookie等）を含むリクエストを許可
Access-Control-Allow-Credentials: true

// プリフライト結果のキャッシュ時間（秒）
Access-Control-Max-Age: 86400

// JS から読み取れるレスポンスヘッダーを指定
Access-Control-Expose-Headers: X-Total-Count, X-Page

// ── クライアント側（fetch）──
fetch('https://api.other.com/data', {
  method:      'POST',
  credentials: 'include',  // Cookie を送信する場合
  headers: { 'Content-Type': 'application/json' },
  body:    JSON.stringify({ key: 'value' }),
});

// credentials: 'include' のとき
// Access-Control-Allow-Origin: * は使えない（具体的なオリジンが必要）
// Access-Control-Allow-Credentials: true が必要
📌 試験頻出：CORS の重要ルール
  • Access-Control-Allow-Origin: * は認証情報付きリクエスト（credentials: 'include'）と併用できない
  • 認証情報を使う場合は具体的なオリジンを指定し、Access-Control-Allow-Credentials: true も必要
  • CORS はブラウザの制限。サーバー間通信（Node.js から Node.js など）には適用されない
  • プリフライトリクエストは OPTIONS メソッドで送信される

3. シンプルリクエストとプリフライトリクエスト

CORS リクエストはシンプルリクエストとプリフライトリクエストの2種類に分かれます。シンプルリクエストは事前確認なしに送信されますが、プリフライトリクエストは本リクエストの前に OPTIONS リクエストで許可を確認します。

シンプルリクエストの条件 
// ── シンプルリクエストになる条件（すべてを満たす場合）──

// ① メソッドが以下のいずれか
//    GET / HEAD / POST

// ② ヘッダーが CORS セーフリストヘッダーのみ
//    Accept
//    Accept-Language
//    Content-Language
//    Content-Type（値に制限あり ↓）

// ③ Content-Type が以下のいずれか（POST の場合）
//    application/x-www-form-urlencoded
//    multipart/form-data
//    text/plain

// ── シンプルリクエストの例 ──
fetch('https://api.other.com/data');  // GET → シンプル

fetch('https://api.other.com/data', {
  method:  'POST',
  headers: { 'Content-Type': 'text/plain' },
  body:    'hello'
});  // POST + text/plain → シンプル

// ── プリフライトが発生する例 ──
fetch('https://api.other.com/data', {
  method:  'PUT',  // PUT → プリフライト
});

fetch('https://api.other.com/data', {
  method:  'POST',
  headers: { 'Content-Type': 'application/json' },  // JSON → プリフライト
});

fetch('https://api.other.com/data', {
  headers: { 'Authorization': 'Bearer token' },  // カスタムヘッダー → プリフライト
});

// ── プリフライトリクエストの流れ ──
// 1. ブラウザが OPTIONS リクエストを送信
//    Origin: https://example.com
//    Access-Control-Request-Method: PUT
//    Access-Control-Request-Headers: Content-Type, Authorization
//
// 2. サーバーが許可を返す
//    Access-Control-Allow-Origin: https://example.com
//    Access-Control-Allow-Methods: PUT, POST, GET
//    Access-Control-Allow-Headers: Content-Type, Authorization
//    Access-Control-Max-Age: 86400
//
// 3. 本リクエストを送信
条件シンプルリクエストプリフライト発生
メソッド: GET✅—
メソッド: HEAD✅—
メソッド: POST + text/plain✅—
メソッド: POST + application/json—✅ Content-Type が非セーフリスト
メソッド: PUT / DELETE / PATCH—✅ メソッドが非セーフリスト
カスタムヘッダー（Authorization 等）—✅ セーフリスト外のヘッダー
credentials: 'include'条件次第サーバー側の設定が必要

4. SSL / HTTPS とセキュリティヘッダー

主要なセキュリティヘッダー 
// ── Strict-Transport-Security（HSTS）──
// HTTPS のみでアクセスさせる
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

// ── Content-Security-Policy（CSP）──
// 読み込めるリソースの出所を制限（XSS 対策）
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com

// ── X-Frame-Options ──
// iframe での埋め込みを制限（クリックジャッキング対策）
X-Frame-Options: DENY          // 埋め込み禁止
X-Frame-Options: SAMEORIGIN    // 同一オリジンのみ許可

// ── X-Content-Type-Options ──
// MIME スニッフィングを防止
X-Content-Type-Options: nosniff

// ── Referrer-Policy ──
// Referer ヘッダーの送信を制御
Referrer-Policy: strict-origin-when-cross-origin

// ── Permissions-Policy（旧 Feature-Policy）──
// ブラウザ機能へのアクセスを制限
Permissions-Policy: geolocation=(), camera=(), microphone=()

// ── CORS と SSL の関係 ──
// HTTPS ページから HTTP へのリクエストは
// Mixed Content として多くのブラウザでブロックされる
// wss:// (WebSocket Secure) は HTTPS ページで必須
⚠️ Mixed Content に注意

HTTPS ページから HTTP リソース（画像・スクリプト・API 等）を読み込もうとすると Mixed Content エラーが発生し、ブラウザにブロックされます。HTTPS サイトでは、すべてのリソースを HTTPS で提供する必要があります。WebSocket も wss://（WebSocket Secure）を使用してください。

5 セキュリティヘッダー解析デモ

各セキュリティヘッダーをクリックすると、詳細な説明・設定例・効果を確認できます。

Strict-Transport-Security
HTTPS 接続を強制する（HSTS）
目的： HTTP でアクセスしても自動的に HTTPS にリダイレクトさせる
設定例：
max-age=31536000 — 1年間 HTTPS を強制
includeSubDomains — サブドメインにも適用
preload — ブラウザの HSTS プリロードリストに登録可能
効果： SSL ストリッピング攻撃を防止
Content-Security-Policy
読み込めるリソースの出所を制限（XSS 対策）
目的： XSS（クロスサイトスクリプティング）攻撃を軽減
主なディレクティブ：
default-src 'self' — 同一オリジンのみ許可
script-src 'nonce-xxx' — nonce 付きスクリプトのみ許可
img-src * — 画像はすべて許可
frame-ancestors 'none' — iframe 埋め込み禁止
効果： インラインスクリプトや外部スクリプトの実行を制限
X-Frame-Options
iframe での埋め込みを制限（クリックジャッキング対策）
目的： クリックジャッキング攻撃を防止
値：
DENY — iframe での埋め込みを完全禁止
SAMEORIGIN — 同一オリジンの iframe のみ許可
ALLOW-FROM uri — 特定の URI のみ許可（非推奨）
注意： 現代のブラウザでは CSP の frame-ancestors が推奨
X-Content-Type-Options
MIME スニッフィングを防止
目的： ブラウザが Content-Type を無視して MIME タイプを推測（スニッフィング）するのを防ぐ
値： nosniff のみ
効果： 悪意のあるファイルが別の MIME タイプとして実行されるのを防止
例: テキストファイルが JavaScript として実行されるのを防ぐ
Referrer-Policy
Referer ヘッダーの送信を制御
目的： クロスオリジンリクエスト時に送信される Referer 情報を制御
主な値：
no-referrer — Referer を送信しない
same-origin — 同一オリジンのみ送信
strict-origin-when-cross-origin — クロスオリジンはオリジンのみ送信（推奨）
unsafe-url — 常に完全な URL を送信（非推奨）
Permissions-Policy
ブラウザ機能へのアクセスを制限
目的： Geolocation・カメラ・マイクなどのブラウザ機能へのアクセスを制限
設定例：
geolocation=() — Geolocation を無効化
camera=(self) — 自サイトのみカメラ許可
microphone=("https://meet.example.com") — 特定サイトのみ許可
旧名称： Feature-Policy（現在は Permissions-Policy に統一）
▶ ヘッダーカードをクリックして詳細を確認してください

4-7 セキュリティモデルの理解を確認しましょう。

QUESTION 1 / 6
基準 URL が https://example.com/page.html のとき、同一オリジンとなる URL はどれですか？
✅ 正解：C
オリジンはプロトコル・ホスト・ポートの3要素で決まります。
A ✗ プロトコルが違う（http vs https）
B ✗ ホストが違う（www.example.com vs example.com）
C ✅ パスが違うだけ → 同一オリジン（プロトコル=https、ホスト=example.com、ポート=443 が一致）
D ✗ ポートが違う（443 vs 8080）
QUESTION 2 / 6
CORS のプリフライトリクエストで使用される HTTP メソッドはどれですか？
✅ 正解：B（OPTIONS）
プリフライトリクエストはブラウザが自動的に OPTIONS メソッドで送信します。
リクエストには以下のヘッダーが含まれます：
• Origin: リクエスト元のオリジン
• Access-Control-Request-Method: 本リクエストで使うメソッド
• Access-Control-Request-Headers: 本リクエストで使うカスタムヘッダー
PREFLIGHT というメソッドは存在しません。
QUESTION 3 / 6
以下のリクエストのうち、CORS のシンプルリクエストになるものはどれですか？
✅ 正解：A
シンプルリクエストの条件：
• メソッドが GET / HEAD / POST のいずれか
• ヘッダーが CORS セーフリスト（Accept・Accept-Language・Content-Language・Content-Type）のみ
• POST の場合、Content-Type が text/plain / application/x-www-form-urlencoded / multipart/form-data のいずれか
A ✅ GET + カスタムヘッダーなし → シンプルリクエスト
B ✗ application/json はセーフリスト外 → プリフライト
C ✗ PUT はセーフリスト外のメソッド → プリフライト
D ✗ Authorization はセーフリスト外のヘッダー → プリフライト
QUESTION 4 / 6
Access-Control-Allow-Origin: * を設定したとき、できないことはどれですか？
✅ 正解：C
Access-Control-Allow-Origin: * は認証情報付きリクエストと併用できません。
credentials: 'include'（Cookie・認証ヘッダーを送信）のリクエストを受け付けるには：
1. Access-Control-Allow-Origin に具体的なオリジンを指定（* 不可）
2. Access-Control-Allow-Credentials: true を設定

これはセキュリティ上の重要な制約です。* は認証情報なしのリクエストのみ許可します。
QUESTION 5 / 6（複数選択）
同一オリジンポリシーによって制限される操作をすべて選んでください。
※ 正しいものをすべてクリックして「確認」を押してください
✅ 正解：A・C・E
A ✅ fetch/XHR によるクロスオリジンリクエストは制限される（CORS で許可が必要）
B ✗ <img> タグでの画像表示は許可される（埋め込みは許可）
C ✅ クロスオリジン iframe の DOM へのアクセスは制限される
D ✗ <script src> でのスクリプト読み込みは許可される（CDN が使えるのはこのため）
E ✅ 別オリジンの localStorage にはアクセスできない

同一オリジンポリシーは「読み取り」を制限します。「埋め込み」（img・script・link・iframe）は基本的に許可されます。
QUESTION 6 / 6（穴埋め）
